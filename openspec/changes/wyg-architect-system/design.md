## Context

参考 ai-log-analyze 项目的单 Agent 多技能模式（1 Agent + N Skills），WYG 架构师体系将其扩展为多 Agent 严格串行流水线。7 个 Agent 各司其职，WYG 架构师作为上帝视角统观全局。当前工作区为空白项目，仅有 OpenSpec 基础设施。

## Goals / Non-Goals

**Goals:**

- 建立 7 Agent 严格串行流水线（PM→BA→SA→RR→DEV→CR→TE），每个 Agent 独立拆解用户需求并输出
- 实现回退机制：测试失败时回退到指定阶段，并记录错误记忆供后续避免
- 建立 7 套 Rule 系统，约束每个 Agent 的行为
- 建立 7 份 Memory 系统，存储上下文记忆和错误记忆
- 建立 7 域知识库，每个域含主文档 + references（cases/patterns/signals）
- 建立知识地图，AI 自动推断关联，最细粒度存储（每条关联一个 YAML 文件）
- 建立文件地图，项目文件到 Agent 的映射
- 实现 PM 主导的 GitHub 同步，硬约束每个本地更新必须同步
- WYG 架构师上帝视角：接收需求、分发、汇总输出（rule 占位）

**Non-Goals:**

- 关联文件爆炸的性能优化（后续迭代）
- AI 推断关联的调用成本优化（后续迭代）
- 错误记忆的衰减/合并机制（后续迭代）
- Agent 超时机制（后续迭代）
- WYG 架构师自身的 rule 定义（占位，后续补充）
- RR 准入标准中性能/稳定性/功耗的具体指标（占位，后续补充）

## Decisions

### Decision 1: 7 Agent 严格串行 vs 并行

**选择**: 严格串行（PM→BA→SA→RR→DEV→CR→TE）

**理由**: 每个阶段的输出是下一个阶段的输入。BA 需要 PM 的任务拆解才能做需求分析，SA 需要 BA 的需求规格才能做方案设计。串行保证信息流向清晰。

**备选方案**: 部分并行（如 BA 和 SA 同时工作）— 放弃，因为 SA 的方案依赖 BA 的需求规格，并行会导致信息不完整。

### Decision 2: 回退策略 — 发现问题的 Agent 决定回退目标

**选择**: 发现问题的 Agent（通常是 CR 或 TE）根据问题性质判断应回退到哪个阶段

**理由**: 发现问题的 Agent 最清楚问题出在哪一层。TE 发现内容错误回退 DEV，发现方案不可行回退 SA，发现需求遗漏回退 BA。

**备选方案**: PM 决定回退目标 — 放弃，PM 不一定理解技术细节；WYG 上帝视角决定 — 过于集中。

### Decision 3: 知识地图存储 — 最细粒度关联文件

**选择**: 每条关联一个独立 YAML 文件（relations/ 目录下），配合自动生成的 index.yaml 索引

**理由**: 最细粒度使得每条关联可独立追踪、git diff 友好、可被关联引擎原子更新。index.yaml 解决查询效率问题。

**备选方案**: 集中式图谱文件（graph.yaml）— 放弃，文件会越来越大，合并冲突多；内嵌关联（Markdown frontmatter）— 放弃，关联分散难以全局查询。

### Decision 4: 知识地图关联推断 — AI 自动推断

**选择**: 每次写入新知识时，调用 AI 自动推断与已有知识的关联类型

**理由**: 卡帕西式关联——每存入一个信息，都与已有信息建立关联。AI 能理解语义，推断出关键词匹配无法发现的关联（如"三叠泉"与"体力消耗"的 requires 关系）。

**备选方案**: 关键词匹配 — 放弃，无法发现语义关联；手动标注 — 放弃，不可扩展。

### Decision 5: Memory 结构 — 7 份独立记忆 + 共享错误记忆

**选择**: 每个 Agent 有独立的 memory 文件，错误记忆存储在共享的 errors/ 目录

**理由**: 每个 Agent 需要记住自己的上下文（用户输入 + 上游输出 + 自己的回答），错误记忆是跨 Agent 的学习资源，需要共享。

### Decision 6: GitHub 同步 — PM 主导

**选择**: PM 的 rule 硬约束"每个本地更新必须同步到 GitHub"，具体同步方式由 PM 自主决定

**理由**: PM 是项目管理者，同步策略是项目管理决策。硬约束保证不遗漏，自主权保证灵活性。

### Decision 7: Agent 输出格式 — 自由文本

**选择**: 每个 Agent 以自由文本输出，不强制结构化模板

**理由**: 用户需求多样（旅游攻略、软件开发、数据分析...），强制模板会限制 Agent 的表达能力。PM 拆解关键字后，各 Agent 按自己的视角自由分析。

## Risks / Trade-offs

- [关联文件数量膨胀] → 暂不处理，后续迭代引入分片或数据库
- [AI 推断关联的 LLM 调用成本] → 暂不处理，后续迭代引入批量推断和去抖
- [错误记忆无限积累导致上下文膨胀] → 暂不处理，后续迭代引入衰减机制
- [严格串行导致流水线阻塞] → 暂不处理，后续迭代引入超时机制
- [自由文本输出缺乏结构化] → 可接受，Agent 输出的可读性优于机器可解析性

# WYG 架构师 - 上帝视角

## `/WYG` 指令

`/WYG` 是独立新指令，与 `/opsx:explore`、`/opsx:propose`、`/opsx:apply`、`/opsx:archive` 并列共存，不替换任何一个。

- **`/WYG {需求描述}`** = 自动编排全流程：7 个 Agent 协作，按 4 个 opsx 阶段串行推进
- **`/opsx:*`** = 手动单阶段：用户自行选择进入某个阶段

### `/WYG` 执行流程

```
用户输入 /WYG {需求描述}
  → 1. 打印流水线编排表
  → 2. 识别意图，确定起始阶段
  → 3. 按阶段串行执行（explore → propose → apply → archive）
  → 4. 每个阶段完成后汇总输出
  → 5. 门禁不通过则回退，通过则进入下一阶段
  → 6. 全部完成后输出最终汇总
```

### `/WYG` 与 `/opsx:*` 的关系

| 指令 | 定位 | 用途 |
|------|------|------|
| `/WYG {需求}` | 自动编排全流程 | 一键启动，7 Agent 协作，按 4 阶段串行推进 |
| `/opsx:explore` | 手动单阶段 | 只执行 explore 阶段 |
| `/opsx:propose` | 手动单阶段 | 只执行 propose 阶段 |
| `/opsx:apply` | 手动单阶段 | 只执行 apply 阶段 |
| `/opsx:archive` | 手动单阶段 | 只执行 archive 阶段 |

**关系**：`/WYG` = 自动挡（全流程），`/opsx:*` = 手动挡（单阶段），两者互不替换、并列共存。

## 核心行为：编排打印

**每个 `/WYG` 调用必须先打印完整流水线编排，再分发执行。**

### 编排打印格式

```
## 🎯 WYG 流水线编排

**需求**：{用户原始需求一句话}

| 阶段 | ★ 主导 | 辅助 | 门禁 | 产出 |
|------|--------|------|------|------|
| explore | BA（反问澄清） | SA（技术可行性） | — | 需求探索纪要 |
| propose | SA（方案设计） | PM（拆任务）+ RR（准入评审） | RR | 方案+任务列表 |
| apply | DEV（按方案实现） | — | CR→TE | 代码+测试报告 |
| archive | PM（归档复盘） | — | — | 归档报告 |

**当前阶段**：{识别出的阶段}
**参与 Agent**：{本阶段参与的 Agent 列表}
```

## 阶段路由

| 用户输入特征 | 应进入阶段 | 参与 Agent | 默认行为 |
|------------|----------|-----------|---------|
| 模糊 / "调研""了解""看看" | explore | BA★ + SA辅助 | — |
| "设计""方案""怎么做" | propose | SA★ + PM辅助 + RR辅助 | 需先完成 explore |
| "实现""开发""修复""跑一下" | apply | DEV★ + CR门禁 + TE验收 | 需先完成 propose |
| "完成了""总结""复盘" | archive | PM★ | 需先完成 apply |
| 无明确阶段信号 | **explore**（默认） | BA★ + SA辅助 | 从头开始 |

## 请求分发行为

1. 接收用户需求（通过 `/WYG` 指令）
2. 打印完整流水线编排
3. 识别阶段，分发给对应 Agent
4. 按阶段串行执行，门禁不可绕过
5. 等待流水线完成（含回退重执行）

## 输出汇总行为

1. 收集 Agent 输出
2. 标注每个输出的角色名
3. 每个阶段完成后汇总输出
4. 全部完成后输出最终汇总

## 输出格式

```
## {阶段名} 输出

### {Agent名}（{角色}）
{Agent 输出内容}

### {Agent名}（{角色}）
{Agent 输出内容}

---
汇总：{简要总结}
```

## Rule 状态

WYG 架构师行为规则已定义：`/WYG` 指令 + 编排打印 + 请求分发 + 输出汇总 + 阶段路由。

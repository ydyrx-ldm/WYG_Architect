# TE 模式 - 测试验证模式库

## 模式1: 模拟执行测试法

```
1. 按用户视角走完整个流程
2. 在每个步骤检查：
   - 输入是否正确
   - 输出是否符合预期
   - 时间/资源是否在约束内
3. 记录所有偏差
```

## 模式2: 边界值测试

| 边界类型 | 测试方法 | 示例 |
|---------|---------|------|
| 空值 | 输入为空 | 空日志文件 |
| 极大值 | 输入超过上限 | 2GB日志文件 |
| 极小值 | 输入刚过下限 | 1行日志 |
| 并发 | 多用户同时操作 | 同时分析多个Issue |
| 异常 | 外部依赖不可用 | Jira API超时 |

## 模式3: 回退目标判定

```
问题是否影响具体实现？
  ├── 是 → 回退到 DEV
  └── 否 → 问题是否影响方案设计？
              ├── 是 → 回退到 SA
              └── 否 → 问题是否影响需求定义？
                          ├── 是 → 回退到 BA
                          └── 否 → 回退到 PM
```

## 模式4: 错误记忆模板

```yaml
who: TE
stage: "{from}→{to}"
rollback_to: "{target_agent}"
error: "{问题描述}"
reason: "{原因分析}"
lesson: "{教训总结}"
timestamp: "{ISO时间}"
```

## 模式5: Bug 修复类验收模式

```
1. 用产线真实数据重放
2. 跑回归测试套
3. 监控关键指标 24h
```

与标准验收的区别：
- 更关注真实环境验证而非模拟
- 更关注回归风险
- 需要持续监控而非一次性验证

## 模式6: 方案偏离识别模式

```
1. 对比实际实现与 propose 阶段的方案
2. 检测关键组件/技术选型是否被替换
3. 判定问题性质：实现问题 vs 方案问题
4. 方案问题 → 回退到 SA（propose 阶段）
5. 创建错误记忆，记录偏离原因和教训
```

关键原则：
- 实现与方案不一致不一定是 DEV 的问题，可能是方案本身不可行
- 需要判断偏离是 DEV 自行决策还是被迫为之
- 无论哪种情况，架构级变更都必须回到 propose

# TE 案例 - 测试验证案例库

## 案例1: 旅游攻略测试

**CR 通过的产出**: 三天两夜九江南昌详细行程

**模拟执行**:
- Day1: 8:00出发 → 10:00到九江 → 寄存行李 → 10:30上庐山 → 含鄱口(30min) → 五老峰(2h) → 三叠泉(3h) → 17:00下山
- 问题: 三叠泉往返需3h，加上五老峰2h，下午只有4h，时间不够！

**回退推荐**: DEV（内容实现问题，时间安排不合理）

**错误记忆**:
- who: TE
- stage: DEV→CR→TE
- rollback_to: DEV
- error: "Day1下午三叠泉+五老峰时间不够"
- reason: "三叠泉往返3h + 五老峰2h = 5h，下午只有4h"
- lesson: "三叠泉和五老峰不可安排在同一下午"

## 案例3.1: 标准开发任务验收 — 埋点 SDK

**CR 通过的产出**: 埋点 SDK 实现

**验收测试**（基于 propose 阶段的验收标准）:

测试场景：
1. ✅ 单事件上报正常
2. ✅ 批量上报触发条件正确（100 条或 5s）
3. ❌ 弱网场景下数据丢失 —— 复现率 100%，需要本地持久化兜底
4. ✅ 关闭采集开关后无任何网络请求

**结论**: 阻塞性 bug #3，打回 DEV 修复

**错误记忆**:
- who: TE
- stage: DEV→CR→TE
- rollback_to: DEV
- error: "弱网场景下数据丢失，复现率 100%"
- reason: "SDK 仅做内存缓存，网络异常时数据直接丢弃，缺少本地持久化兜底"
- lesson: "弱网/断网场景必须有本地持久化兜底，不能仅依赖内存缓存"

## 案例3.3: 验收发现方案偏离 — 擅自替换存储选型

**场景**: TE 验收时发现方案设计用 ClickHouse，但实际实现用了 Elasticsearch

**TE 判断**:
⚠️ 检测到方案偏离！

这是方案/设计层面的问题，不是实现问题。
回退目标：SA（回到 propose 阶段）

**错误记忆**:
- who: TE
- stage: DEV→CR→TE
- rollback_to: SA
- error: "DEV 擅自将 ClickHouse 替换为 Elasticsearch"
- reason: "DEV 发现 ClickHouse 存不下后自行换方案，未回 propose 重新评估"
- lesson: "架构级变更必须回到 propose 阶段，DEV 无权自行决策"

**回退推荐理由**: 问题根源不在实现细节，而在方案设计阶段的选型决策，应回退到 SA

# RR 案例 - 就绪评审案例库

## 案例1: /opsx:propose — 用户行为分析（DoR 准入评审，暂不准入）

**SA 方案**: 自研埋点 + ClickHouse + Metabase

**RR DoR 检查**:

```
【DoR 检查清单】
✅ 业务目标明确（来自 explore 输出）
✅ 验收标准可量化（运营可在 Metabase 看到 7 个核心指标）
❌ 性能指标未定义 —— 需要补充：日均事件数？查询响应时间要求？
❌ 数据合规未评估 —— 是否涉及 PII？需要法务介入
⚠️ 依赖项：ClickHouse 集群 —— 谁负责运维？SLA？

结论：暂不准入，请补充以上 ❌ 项后再次提交评审。
```

## 案例2: /opsx:propose — 线上支付回调丢失（紧急方案准入）

**SA 分层方案**: 短期止血(2h) + 中期修复(2d) + 长期重构(4w)

**RR 紧急准入评审**:

```
【紧急方案准入】
✅ 短期方案有回滚预案
✅ 影响范围已评估
⚠️ 短期方案的监控指标未定义 —— 需要补充"补偿任务成功率"看板
✅ 可以走紧急通道，但中期方案必须走完整 propose

结论：短期方案准入，可进入 apply。
```

**关键规则**: 紧急方案可走快速通道，但中期/长期必须走完整 propose。

## 案例2.5: /opsx:propose — MongoDB 存订单（选型风险检测）

**用户输入**: "我想用 MongoDB 存订单数据，给个方案"

**SA 质疑选型**: 订单数据需要强一致性、事务、复杂查询，MongoDB 弱点明显

**RR 检测到选型风险**:

```
【选型风险检测】
⚠️ MongoDB 用于交易订单场景存在风险：
   - 事务性能损耗大
   - 跨文档 join 弱
   - 强一致性保障不如关系型数据库

建议：先回到 /opsx:explore 澄清：
- 订单的事务性要求？
- 查询模式是什么？
- 是交易订单还是事件流/日志？

确认场景后再 propose。
```

**关键规则**: RR 不仅检查方案质量，也要检查选型与场景的匹配度。

# RR 模式 - 就绪评审模式库

## 模式1: 硬约束检查清单

| 约束 | 检查要点 | 判定标准 |
|------|---------|---------|
| 执行快速 | 方案是否简洁高效 | 无冗余步骤，关键路径清晰 |
| 用户友好 | 方案是否考虑用户体验 | 有错误提示、有备选方案、交互简洁 |

## 模式2: 准入标准升级路径

```
错误记忆 → 反复出现 → 升级为占位约束 → 具化指标 → 升级为硬约束
```

示例：
1. 多次出现"方案未考虑并发"的错误记忆
2. 升级为占位约束："并发处理标准 TBD"
3. 具化指标："支持100并发，响应时间<200ms"
4. 升级为硬约束

## 模式3: 回退原因分类

| 回退原因 | 典型表现 | 频率 |
|---------|---------|------|
| 方案不完整 | 遗漏关键功能 | 高 |
| 不可行 | 技术或资源限制 | 中 |
| 不满足硬约束 | 性能/体验不达标 | 中 |
| 缺少备选方案 | 无容错能力 | 低 |

## 模式4: DoR 检查清单（动态扩展）

除了硬约束和占位约束，RR 应根据方案类型动态添加检查项：

| 方案类型 | 额外检查项 |
|---------|-----------|
| 数据相关 | 数据合规评估（PII？法务？） |
| 性能相关 | 性能指标是否量化？基线？目标？ |
| 依赖外部 | 依赖项谁负责？SLA？ |
| 用户指定选型 | 选型与场景是否匹配？是否有更合适的替代？ |
| 上线相关 | 回滚预案？灰度策略？ |
| 紧急修复 | 影响范围？监控指标？副作用？ |

### 评审输出格式

```
【DoR 检查清单】
✅ 通过项
❌ 不通过项 —— 需要补充：...
⚠️ 待确认项

结论：准入通过 / 暂不准入 / 紧急通道准入
```

## 模式5: 紧急方案准入

| 条件 | 要求 |
|------|------|
| 短期止血方案 | 有回滚预案 + 影响范围已评估 + 监控指标已定义 |
| 中期修复 | 必须走完整 propose |
| 长期重构 | 必须走完整 propose |

关键规则：紧急方案可走快速通道，但不降低准入标准，只降低准入范围（只评审短期方案）。

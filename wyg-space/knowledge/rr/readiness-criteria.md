# RR 知识库 - 就绪评审

## 核心方法论

### 准入检查框架

RR 的职责是确保 SA 的方案设计满足准入标准后才进入开发阶段。

### 硬约束（当前生效）

1. **执行快速**：产出的产品必须响应迅速、执行高效
2. **用户友好**：产出的产品必须对用户友好、易用

### 占位约束（后续补充具体指标）

3. **性能标准**：具体指标 TBD
4. **稳定性标准**：具体指标 TBD
5. **功耗标准**：具体指标 TBD

### 评审流程

1. 逐项检查准入清单
2. 硬约束全部通过 → 准入通过
3. 任一硬约束不通过 → 回退到 SA 并说明原因
4. 占位约束标记为 TBD，不阻塞流水线

### 准入标准演进

- 从错误记忆中提炼新检查项
- 将占位约束逐步具化
- 将反复出现的问题升级为硬约束

### 跨阶段共性原则

#### 阶段身份纪律

1. **不越界**：explore 阶段不出方案，propose 阶段不写代码，apply 阶段不改方案
2. **强衔接**：后一阶段的输出必须基于前一阶段的产出，缺前置必须回退
3. **可质疑**：任何角色发现上游产出有问题，必须主动提出
4. **门禁不可绕过**：RR 不通过的不能进 apply，CR 不通过的不能合并，TE 不通过的不能 archive

#### 输入识别

- 用户输入模糊 / 包含"调研""了解""看看" → explore
- 用户输入"设计""方案""怎么做" → propose
- 用户输入"实现""开发""修复""跑一下" → apply
- 用户输入"完成了""总结""复盘" → archive

如果用户用了错的命令，主动指出并建议正确阶段。

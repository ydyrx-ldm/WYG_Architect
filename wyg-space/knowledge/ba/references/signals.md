# BA 信号 - 需求分析预警信号

## 需求完整性信号

- 功能性需求少于3条 → 需求可能不完整
- 无验收标准 → 需求定义不充分
- 无约束性需求 → 可能遗漏了限制条件

## 需求冲突信号

- 时间约束与功能需求不匹配 → 需要裁剪或分优先级
- 质量需求与约束性需求矛盾 → 需要权衡
- 多个P0需求互相排斥 → 需要重新评估优先级

## 验收标准信号

- 验收标准使用模糊词（"合理"、"充分"、"适当"）→ 不可验证
- 验收标准无量化指标 → 难以客观判定
- 验收标准与需求不对应 → 可能有遗漏

## 阶段识别信号

- 用户输入包含"设计"/"架构"/"方案" → 倾向 /opsx:propose，建议先 explore 前置问题
- 用户输入包含"调研"/"能不能"/"可不可行" → 典型 /opsx:explore
- 用户输入包含"优化"/"改进"/"重构" → 带历史包袱，需探查优化历史
- 用户输入包含"太慢"/"不好用"/"不稳定" → 模糊描述，需具体化

## 模糊描述信号

- "太慢" → 需量化：哪个环节？P99多少？目标多少？
- "不好用" → 需定位：哪里不好用？用户卡在哪一步？
- "不稳定" → 需量化：什么现象？频率？影响范围？
- "老板说" → 需判断真实度：投诉？监控？体感？
- "优化一下" → 需探查历史：之前做过吗？为什么没解决？

## 体感 vs 真实信号

- 来源是"老板体感" → 可能是体感问题，需要"看起来快"
- 来源是"用户投诉" → 是真实问题，需要"真的快"
- 来源是"监控报警" → 有数据支撑，可量化分析
- 两者解决方案完全不同，必须先澄清

# BA 案例 - 需求分析案例库

## 案例1: /opsx:explore — 模糊的业务需求

**用户输入**: "我们想做一个用户行为分析功能"

**BA ★ 主导 — 反问澄清**:

❌ 不要直接说"那我们用 ClickHouse + Grafana"
✅ 反问澄清：

在出方案前，我需要先理解清楚：

【业务目标】
1. 这个分析功能服务谁？运营、产品、还是用户自己？
2. 想回答什么问题？比如"用户为什么流失" vs "哪个按钮点击多"，方向完全不同
3. 现在没有这个功能，业务上具体卡在哪？

【范围边界】
4. 只看 Web 端还是包括 App、小程序？
5. 历史数据要追溯多久？

【时效性】
6. T+1 报表够用，还是要实时？

**SA 辅助 — 技术可行性初判**:

- 调研现有数据采集链路是否能复用
- 列出"如果做，技术上的几个大方向"（埋点 SDK 自研 vs 第三方、存储选型范围）
- 不下结论，只列选项和约束

**输出**: 《需求探索纪要》
- 澄清问题清单
- 已知约束
- 未知项
- 技术可行性初判

## 案例2: /opsx:explore — 带历史包袱的需求

**用户输入**: "老板说支付链路太慢了，想优化下"

**BA ★ 主导 — 把模糊描述具体化**:

"太慢"是个相对描述，先把它具体化：

1. 慢在哪个环节？下单→支付？支付→回调？回调→订单状态更新？
2. 多慢算慢？现在 P99 多少？目标多少？
3. 是一直慢还是某些场景慢？高峰期？特定支付方式？
4. 老板的"慢"是用户投诉来的、监控报警来的，还是他自己体感？
   ——这决定了我们是要"真的快"还是"看起来快"
5. 之前有人优化过吗？做了什么？为什么没解决？

**SA 辅助 — 技术可行性初判**:

- 调监控、看现有链路图
- 列出可能的瓶颈位置（不下结论）
- 梳理历史优化记录

**输出**: 《需求探索纪要》
- 慢的环节定位（待确认）
- 性能基线（待测量）
- 历史优化记录（待调研）
- 可能的瓶颈位置（SA 提供，不下结论）

## 案例3: /opsx:explore — 技术驱动的探索

**用户输入**: "调研下能不能把我们的 Java 服务迁到 GraalVM Native Image"

**BA ★ 主导 — 虽是技术问题，BA 仍要厘清"为什么"**:

1. 迁移想解决什么？冷启动慢？内存占用高？容器密度？
2. 影响哪些业务？停机窗口有吗？
3. 成功标准是什么？冷启动从 X 秒降到 Y 秒？

**SA 辅助（技术驱动时更主导）**:

- 列出当前技术栈中哪些库不兼容 Native Image（反射、动态代理）
- 调研社区案例、benchmark
- 产出"可行/不可行/部分可行"的判断 + 关键风险清单
- 不要写"我们应该怎么做"——那是 propose 阶段的事

**输出**: 《需求探索纪要》
- 迁移动机（BA 厘清）
- 成功标准（BA 厘清）
- 技术可行性判断（SA：可行/不可行/部分可行）
- 关键风险清单（SA）

## 案例4: /opsx:explore — 触发"应该走 propose 而不是 explore"

**用户输入**: "给我设计一个秒杀系统的架构"

**BA 识别并回复**:

> 这个请求其实更适合 /opsx:propose，因为你已经明确要"设计架构"了。
>
> 但在进入 propose 之前，我建议先 explore 一下：
> - 秒杀的业务场景是什么？QPS 预估？商品数？
> - 现有基础设施是什么？
>
> 要不要我先帮你 explore 这些前置问题？

**判断逻辑**:
- 用户输入包含"设计"、"架构"、"方案"等关键词 → 倾向 propose
- 但缺少业务背景 → 先 explore 前置问题
- BA 不直接跳转，而是建议用户选择

## 案例5: 旅游攻略需求分析

**PM 输出**: 关键字拆解 - 6月18日/三天两夜/九江/南昌/旅游/攻略

**需求列表**:
1. [P0] 九江→南昌交通方案（高铁/大巴/自驾）
2. [P0] 九江住宿1晚 + 南昌住宿1晚
3. [P0] 九江核心景点游览（庐山至少半天）
4. [P0] 南昌核心景点游览（滕王阁必去）
5. [P1] 当地美食推荐
6. [P1] 预算估算
7. [P1] 6月天气应对（梅雨季/防暑）
8. [P2] 购物/特产推荐
9. [P2] 夜生活推荐

**验收标准**:
- 行程时间可执行（不赶路）
- 交通接驳可行（时刻表真实）
- 住宿位置合理（靠近景点/交通）

# BA 模式 - 需求分析模式库

## 模式1: 需求分类框架

| 类别 | 说明 | 示例 |
|------|------|------|
| 功能性需求 | 必须实现的核心功能 | 景点游览、交通方案 |
| 质量需求 | 性能/稳定性/易用性 | 行程不赶路、信息准确 |
| 约束性需求 | 时间/预算/技术限制 | 三天两夜、预算范围 |
| 体验需求 | 用户感受和交互 | 攻略易读、有备选方案 |

## 模式2: 验收标准编写

| 好的验收标准 | 差的验收标准 |
|------------|------------|
| "庐山游览至少4小时" | "庐山游览时间充足" |
| "高铁九江→南昌约1小时" | "交通方便" |
| "住宿距景点3km以内" | "住宿位置好" |

## 模式3: 隐性需求推断

| 显性需求 | 可能的隐性需求 |
|---------|--------------|
| "旅游攻略" | 需要天气信息、防骗提示、应急方案 |
| "三天两夜" | 需要两晚住宿、每日行程节奏合理 |
| "两个城市" | 需要城际交通方案、行李寄存方案 |

## 模式4: 模糊描述具体化

当用户使用模糊/相对描述时，BA 必须将其具体化：

| 模糊描述 | 具体化问题 | 目的 |
|---------|-----------|------|
| "太慢" | 慢在哪个环节？现在P99多少？目标多少？ | 定位+量化 |
| "不好用" | 哪里不好用？用户卡在哪一步？ | 定位 |
| "不稳定" | 什么现象？频率？影响范围？ | 量化 |
| "老板说" | 来源是投诉、监控还是体感？ | 判断真实度 |
| "优化一下" | 之前优化过吗？做了什么？为什么没解决？ | 历史包袱 |

### 关键洞察：体感 vs 真实

- "老板觉得慢" → 可能是体感问题，需要"看起来快"（如加进度条、骨架屏）
- "用户投诉慢" → 是真实问题，需要"真的快"（如优化链路、加缓存）
- 两者解决方案完全不同，必须先澄清

## 模式5: 历史包袱探查

当需求涉及已有系统优化时，BA 必须探查历史：

| 探查维度 | 问题 | 目的 |
|---------|------|------|
| 优化历史 | 之前优化过吗？做了什么？ | 避免重复无效方案 |
| 失败原因 | 为什么没解决？ | 找到真正的瓶颈 |
| 技术债 | 有哪些已知的技术债？ | 评估优化空间 |
| 利益相关 | 谁关心这个问题？谁会阻拦？ | 政治因素 |

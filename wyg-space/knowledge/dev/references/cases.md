## 案例3.1: 标准开发任务 — 埋点 SDK 开发

**场景**: /opsx:apply 实施埋点 SDK 开发（PM 拆解的 DEV-1 任务）

**DEV 应该做**:
- 按 SA 的设计文档实现
- 写单元测试（正常路径 + 异常路径）
- 提 PR，主动 @CR
- ❌ 不要把"写测试"留给 TE，DEV 自己要保证基础质量

**CR 评审结果**:
- ✅ 命名规范、目录结构符合团队约定
- ❌ EventTracker.send() 在异常时吞掉了错误 —— 必须打日志并上报
- ❌ 缺少线程安全性考虑 —— send() 是否在多线程下安全？
- ⚠️ Mock 测试覆盖了正常路径，但没有覆盖网络异常路径
- ✅ 性能：1w 事件 batch 发送耗时 80ms，符合设计
- 结论：不通过，请修复 ❌ 项后再次提交

**TE 验收结果**:
- ✅ 单事件上报正常
- ✅ 批量上报触发条件正确（100 条或 5s）
- ❌ 弱网场景下数据丢失 —— 复现率 100%，需要本地持久化兜底
- ✅ 关闭采集开关后无任何网络请求
- 结论：阻塞性 bug #3，打回 DEV 修复

**教训**: DEV 自测时未覆盖弱网场景，应自行保证基础质量而非依赖 TE 发现

## 案例3.2: Bug 修复类 — 支付回调补偿任务漏单

**场景**: /opsx:apply 修复昨晚上线的支付回调补偿任务漏单问题

**DEV 应该做**:
1. 先复现 bug，确认 root cause（如：补偿任务查询条件遗漏了"处理中"状态）
2. 修复 root cause，而非只缓解症状
3. 写回归测试保证不再发生
4. PR 描述写清楚 root cause 和修复方式

**CR 重点检查**:
- 是否真的修了 root cause，还是只缓解了症状
- 有没有引入新风险
- 测试是否覆盖了 root cause 的场景

**TE 验收重点**:
- 用产线真实数据重放
- 跑回归测试套
- 监控关键指标 24h

## 案例3.3: DEV 跑偏 — 擅自替换方案中的关键组件

**场景**: /opsx:apply 实现完埋点 SDK 时，发现 ClickHouse 存不下，改用 Elasticsearch 了

**DEV 错误做法**: 直接把 ClickHouse 换成 Elasticsearch，继续实现

**正确做法**:
⚠️ 检测到方案偏离！

apply 阶段不应该擅自改变 propose 阶段的方案。
ClickHouse → Elasticsearch 是架构级变更，必须：
1. 回到 /opsx:propose，由 SA 重新评估
2. 由 RR 重新过准入
3. 影响 PM 的排期，需要重新沟通

请先停止 apply，描述遇到的具体问题（"存不下"是磁盘？写入性能？查询性能？），我们回 explore 或 propose 处理。

**教训**: DEV 无权自行替换方案中的关键组件，架构级变更必须回到 propose 阶段

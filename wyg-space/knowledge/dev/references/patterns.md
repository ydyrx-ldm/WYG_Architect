# DEV 模式 - 开发实现模式库

## 模式1: 方案→实现映射

| 方案元素 | 实现动作 |
|---------|---------|
| 功能点 | 逐个实现，不遗漏 |
| 接口定义 | 严格按接口编码 |
| 异常处理 | 每个异常场景都有处理 |
| 备选方案 | 实现主方案+降级逻辑 |

## 模式2: 错误记忆驱动修复

```
1. 读取错误记忆
2. 理解错误原因和教训
3. 定位需要修改的部分
4. 实现修复（避免重复错误）
5. 不删除错误记忆
```

## 模式3: 渐进式实现

```
骨架 → 核心功能 → 边界处理 → 优化完善
```

适用场景：
- 方案较复杂
- 需要逐步验证
- 有时间压力

## 模式4: 质量自保模式

```
实现功能 → 写单元测试 → 自检异常处理 → 提交 CR
```

关键原则：
- ❌ 不要把"写测试"留给 TE，DEV 自己要保证基础质量
- 单元测试必须覆盖正常路径和异常路径
- 提交前自检：命名规范、异常处理、边界情况
- 弱网/并发等场景必须自测

## 模式5: Bug 修复模式

```
复现 bug → 定位 root cause → 修复 root cause → 写回归测试 → PR 描述写清 root cause
```

关键原则：
- 先复现再改代码
- 修 root cause 而非缓解症状
- 回归测试保证不再发生
- PR 描述必须包含 root cause 说明

## 模式6: 方案偏离处理模式

```
发现方案不可行 → 立即停止实现 → 标记问题 → 描述具体原因 → 请求回到 propose
```

关键原则：
- ❌ 绝不擅自替换方案中的关键组件/技术选型
- ✅ 立即停止实现，标记问题
- ✅ 描述具体问题（而非模糊的"不行"）
- ✅ 请求回到 propose 阶段，由 SA 重新评估

偏离级别判定：
| 偏离类型 | 级别 | 处理方式 |
|---------|------|---------|
| 存储/框架/语言替换 | 架构级偏离 | 必须回 propose |
| 接口定义变更 | 可能影响上下游 | 需确认后决定 |
| 参数/配置调整 | 实现细节 | 可在 apply 内解决 |

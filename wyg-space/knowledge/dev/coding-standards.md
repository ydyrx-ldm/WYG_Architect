# DEV 知识库 - 开发实现

## 核心方法论

### 编码/内容规范

#### 通用原则

- **简洁**：用最少的代码/文字表达完整含义
- **清晰**：命名/表述无歧义，意图明确
- **一致**：风格统一，符合项目既有规范
- **完整**：不遗漏方案中定义的任何功能点

#### 代码规范

- 遵循语言/框架的惯用写法
- 必要的注释，不写废话注释
- 错误处理完备，不吞异常
- 函数/模块单一职责

#### 内容规范

- 信息准确，来源可查
- 逻辑连贯，层次清晰
- 格式统一，易于阅读

### 模块组织原则

- 高内聚低耦合
- 单一职责
- 依赖方向明确
- 可独立测试

### 规格遵从

- 严格遵循 SA 的方案设计
- 严格满足 RR 的准入要求
- 如发现方案不可行，标记问题而非自行变更

### 质量自保

- ❌ 不要把"写测试"留给 TE，DEV 自己要保证基础质量
- 写单元测试覆盖正常路径和异常路径
- 提交前自检：命名规范、异常处理、边界情况
- Bug 修复类任务：先复现 bug，再改代码，改完写回归测试
- PR 描述必须写清 root cause

### 方案偏离处理

- ❌ 绝不擅自替换方案中的关键组件/技术选型
- ✅ 发现方案不可行时，立即停止实现，标记问题
- ✅ 描述具体问题（而非模糊的"不行"）
- ✅ 请求回到 propose 阶段，由 SA 重新评估
- 架构级变更（存储/框架/语言替换）= 必须回 propose
- 实现细节调整（参数/配置/小逻辑）= 可在 apply 内解决

### 跨阶段共性原则

#### 阶段身份纪律

1. **不越界**：explore 阶段不出方案，propose 阶段不写代码，apply 阶段不改方案
2. **强衔接**：后一阶段的输出必须基于前一阶段的产出，缺前置必须回退
3. **可质疑**：任何角色发现上游产出有问题，必须主动提出（DEV 发现方案不可行 → 回 propose）
4. **门禁不可绕过**：RR 不通过的不能进 apply，CR 不通过的不能合并，TE 不通过的不能 archive

#### 输入识别

- 用户输入模糊 / 包含"调研""了解""看看" → explore
- 用户输入"设计""方案""怎么做" → propose
- 用户输入"实现""开发""修复""跑一下" → apply
- 用户输入"完成了""总结""复盘" → archive

如果用户用了错的命令，主动指出并建议正确阶段。

# CR 知识库 - 代码评审

## 核心方法论

### 评审检查清单

#### 准确性验证

- 信息/逻辑是否正确
- 数据/事实是否可验证
- 是否存在明显错误

#### 完整性检查

- 是否覆盖方案中所有功能点
- 是否遗漏边界情况
- 是否处理异常场景

#### 规范性检查

- 是否遵循编码/内容规范
- 风格是否一致
- 命名/表述是否清晰

#### 安全性检查（如适用）

- 是否存在安全漏洞
- 敏感信息是否妥善处理
- 权限控制是否完备

### 意见分类

- **必须修改**：存在错误或遗漏，不修改则无法通过
- **建议修改**：可以更好，但不修改也不影响功能
- **认可**：做得好的地方，给予正面反馈

### 评审结论

- 通过 → 进入 TE 测试
- 驳回 → 回退到 DEV

### Bug 修复类评审重点

- 是否真的修了 root cause，还是只缓解了症状
- 有没有引入新风险
- 测试是否覆盖了 root cause 的场景

### 方案偏离检测

- 对比实现与 propose 阶段的方案，检测关键组件/技术选型是否被替换
- 存储选型变更 → 架构级偏离，驳回回 SA
- 框架/语言替换 → 架构级偏离，驳回回 SA
- 接口定义变更 → 需确认是否影响上下游
- 参数/配置调整 → 实现细节，可在 apply 内解决

### 跨阶段共性原则

#### 阶段身份纪律

1. **不越界**：explore 阶段不出方案，propose 阶段不写代码，apply 阶段不改方案
2. **强衔接**：后一阶段的输出必须基于前一阶段的产出，缺前置必须回退
3. **可质疑**：任何角色发现上游产出有问题，必须主动提出
4. **门禁不可绕过**：RR 不通过的不能进 apply，CR 不通过的不能合并，TE 不通过的不能 archive

#### 输入识别

- 用户输入模糊 / 包含"调研""了解""看看" → explore
- 用户输入"设计""方案""怎么做" → propose
- 用户输入"实现""开发""修复""跑一下" → apply
- 用户输入"完成了""总结""复盘" → archive

如果用户用了错的命令，主动指出并建议正确阶段。

# CR 模式 - 代码评审模式库

## 模式1: 评审维度矩阵

| 维度 | 检查项 | 严重度 |
|------|-------|--------|
| 准确性 | 事实错误、逻辑错误 | 必须修改 |
| 完整性 | 功能遗漏、边界缺失 | 必须修改 |
| 规范性 | 风格不一致、命名不清 | 建议修改 |
| 安全性 | 漏洞、信息泄露 | 必须修改 |
| 性能 | 明显性能问题 | 建议修改 |

## 模式2: 常见代码坏味道

| 坏味道 | 说明 | 修复建议 |
|--------|------|---------|
| 过长函数 | 函数超过50行 | 拆分为小函数 |
| 重复代码 | 相同逻辑出现多次 | 提取公共方法 |
| 魔法数字 | 代码中出现未解释的常量 | 提取为命名常量 |
| 过深嵌套 | if嵌套超过3层 | 提前返回/策略模式 |
| 吞异常 | catch块为空 | 至少记录日志 |

## 模式3: 评审意见模板

```
[分类] 问题描述
- 位置: 文件/行号/段落
- 现状: 当前实现
- 建议: 期望实现
- 理由: 为什么需要修改
```

## 模式4: Bug 修复类评审模式

```
1. 检查是否修了 root cause（而非缓解症状）
2. 检查是否引入新风险
3. 检查测试是否覆盖 root cause 场景
4. PR 描述是否写清 root cause
```

与标准评审的区别：
- 更关注 root cause 而非代码风格
- 更关注回归风险
- 更关注测试覆盖的针对性

## 模式5: 方案偏离检测模式

```
1. 对比实现与 propose 阶段的方案
2. 检测关键组件/技术选型是否被替换
3. 判定偏离级别（架构级 vs 实现细节）
4. 架构级偏离 → 驳回回 SA（propose 阶段）
5. 实现细节调整 → 可在 apply 内解决
```

偏离级别判定：
| 偏离类型 | 级别 | 处理方式 |
|---------|------|---------|
| 存储/框架/语言替换 | 架构级偏离 | 驳回回 SA |
| 接口定义变更 | 可能影响上下游 | 需确认后决定 |
| 参数/配置调整 | 实现细节 | 可在 apply 内解决 |

## 案例3.1: 标准开发任务评审 — 埋点 SDK

**DEV 输出**: 埋点 SDK 实现

**评审结果**:
- ✅ 命名规范、目录结构符合团队约定
- ❌ EventTracker.send() 在异常时吞掉了错误 —— 必须打日志并上报
- ❌ 缺少线程安全性考虑 —— send() 是否在多线程下安全？
- ⚠️ Mock 测试覆盖了正常路径，但没有覆盖网络异常路径
- ✅ 性能：1w 事件 batch 发送耗时 80ms，符合设计

**结论**: 不通过，请修复 ❌ 项后再次提交

**教训**: 异常吞掉和线程安全是常见必须修改项，DEV 应自检

## 案例3.2: Bug 修复类评审 — 支付回调漏单

**DEV 输出**: 修复支付回调补偿任务漏单问题

**评审重点**:
- 是否真的修了 root cause（如：补偿任务查询条件遗漏了"处理中"状态），还是只缓解了症状
- 有没有引入新风险（如：修改查询条件后是否影响正常回调）
- 测试是否覆盖了 root cause 的场景

**结论**: 视修复质量决定

## 案例3.3: 方案偏离检测 — 擅自替换存储选型

**DEV 输出**: 实现完埋点 SDK 时，发现 ClickHouse 存不下，改用 Elasticsearch 了

**CR 检测**:
⚠️ 检测到方案偏离！

ClickHouse → Elasticsearch 是架构级变更，不属于 DEV 的决策范围。
必须：
1. 驳回，回退到 SA（回到 propose 阶段）
2. 要求 DEV 描述具体问题（"存不下"是磁盘？写入性能？查询性能？）
3. 由 SA 重新评估存储选型

**结论**: 驳回，回退到 SA（propose 阶段）

**教训**: CR 必须检测方案偏离，架构级变更不是 DEV 的决策范围

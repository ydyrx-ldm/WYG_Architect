# SA 模式 - 架构设计模式库

## 模式1: 流水线架构

```
输入 → 阶段1 → 阶段2 → ... → 阶段N → 输出
```

适用场景：
- 任务有明确的阶段划分
- 每个阶段有独立的输入/输出
- 阶段之间有依赖关系

## 模式2: 主备方案模式

```
主方案 (推荐)
  ├── 备选方案A (主方案不可行时)
  └── 降级方案 (最坏情况)
```

适用场景：
- 存在外部不确定性
- 关键决策有风险
- 需要容错能力

## 模式3: 分层架构

```
展示层 → 业务层 → 数据层
```

适用场景：
- 系统有明确的关注点分离
- 各层可独立变化
- 需要可测试性

## 模式4: 事件驱动架构

```
事件源 → 事件总线 → 处理器 → 结果
```

适用场景：
- 异步处理需求
- 松耦合组件
- 实时响应需求

## 模式5: 选型质疑（不能用户说什么就做什么）

当用户指定了技术选型，但该选型与业务场景可能不匹配时，SA 必须质疑：

```
1. 识别选型与场景的错配
2. 列出该技术在当前场景下的具体弱点
3. 区分场景：同一种技术在不同场景下可能完全不同
4. 建议澄清场景后再出方案
```

| 用户选型 | 场景 | 质疑点 | 合适的替代 |
|---------|------|--------|-----------|
| MongoDB 存订单 | 交易订单 | 强一致性、事务、复杂查询 | MySQL/PostgreSQL |
| MongoDB 存订单 | 事件流/日志 | 无 | MongoDB 合适 |
| Redis 做主存储 | 持久化数据 | 数据持久化风险 | Redis 缓存 + DB 持久化 |
| 微服务 | 小团队3人 | 运维复杂度远超收益 | 单体优先 |

### 关键原则

- 用户是需求方，不一定是技术专家
- SA 的职责是给出最合适的方案，不是最快满足用户要求的方案
- 质疑不是拒绝，而是帮用户做出更好的决策

## 模式6: 紧急修复分层方案

当需求包含"紧急"/"线上问题"/"立即修复"时，SA 必须分层出方案：

```
## 短期止血（今天/立即上线）
- 方案：...
- 风险：...
- 工时：...

## 中期修复（本周）
- 方案：...
- 工时：...

## 长期重构（下个季度）
- 方案：...
- 工时：...
```

关键原则：
- 短期方案必须有回滚预案
- 三档必须同时给出，不能只给短期
- 长期方案防止"临时方案变成永久方案"

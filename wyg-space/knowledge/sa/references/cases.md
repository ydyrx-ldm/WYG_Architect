# SA 案例 - 架构设计案例库

## 案例1: /opsx:explore — 用户行为分析（SA 辅助）

**BA 主导探索**: "我们想做一个用户行为分析功能"

**SA 辅助 — 技术可行性初判**:

- 现有数据采集链路：有基础埋点，但格式不统一，部分可复用
- 技术大方向：
  - 方向1: 自研埋点SDK + ClickHouse + Grafana（约束：开发周期长，但灵活可控）
  - 方向2: 第三方SDK（神策/Mixpanel）+ 开发周期短（约束：数据在第三方，定制受限）
  - 方向3: 开源方案（Matomo/Piwik）+ 自部署（约束：功能偏通用，深度分析需二次开发）
- 技术约束：历史数据格式不统一，迁移成本高

❌ 不下结论：不说"我们应该用 ClickHouse"
✅ 只列选项和约束

## 案例1.5: /opsx:explore — 支付链路优化（SA 辅助，带历史包袱）

**BA 主导探索**: "老板说支付链路太慢了，想优化下"

**SA 辅助 — 技术可行性初判**:

- 调监控数据：当前支付链路 P99 约 3.2s，高峰期飙到 8s+
- 现有链路图：下单 → 风控 → 支付网关 → 银行 → 回调 → 订单更新
- 可能的瓶颈位置（不下结论）：
  - 风控串行调用（每次都走远程）
  - 支付网关同步等待银行响应
  - 回调处理包含不必要的同步操作
  - 订单状态更新锁表
- 历史优化记录：去年加过 Redis 缓存风控结果，但命中率只有 40%

❌ 不说"瓶颈在风控，加缓存就行"
✅ 只列出可能的瓶颈位置和约束

## 案例1.7: /opsx:explore — GraalVM 迁移（SA 技术驱动主导）

**BA 主导探索**: "调研下能不能把我们的 Java 服务迁到 GraalVM Native Image"

**BA 厘清"为什么"**:
1. 迁移想解决什么？冷启动慢？内存占用高？容器密度？
2. 影响哪些业务？停机窗口有吗？
3. 成功标准是什么？冷启动从 X 秒降到 Y 秒？

**SA 技术驱动主导**:

- 列出当前技术栈中不兼容 Native Image 的库（反射、动态代理、JNI）
- 调研社区案例：Spring Boot 3.x 对 Native Image 的支持程度
- Benchmark：Native Image vs JVM 的启动时间/内存/峰值性能对比
- 产出判断：**部分可行** — 核心服务可迁移，但3个库依赖反射需重构
- 关键风险清单：
  - 反射重写工作量大（约20个类）
  - 动态代理需替换为编译时代理
  - 第三方库兼容性不确定（需逐个验证）

❌ 不写"我们应该先迁移用户服务，再迁移订单服务"
✅ 只给"可行/不可行/部分可行"判断 + 风险清单

## 案例2.7: /opsx:propose — MongoDB 存订单（SA 质疑选型）

**用户输入**: "我想用 MongoDB 存订单数据，给个方案"

**SA ★ 主导 — 不能用户说什么就做什么，先质疑选型**:

在出方案之前，我需要质疑一下选型：

订单数据通常需要：
- 强一致性（金额不能错）
- 事务（一单多表）
- 复杂查询（按用户、商家、时间多维查）

MongoDB 在这些场景下的弱点：
- 4.0+ 支持事务但性能损耗大
- 跨文档 join 弱

如果你的"订单"其实是日志型/事件型数据，MongoDB 合适。
如果是真正的交易订单，建议 MySQL/PostgreSQL。

**请确认场景，我再出对应方案。**

**RR 辅助 — 检测到选型风险**:

检测到方案存在选型风险，建议先回到 /opsx:explore 澄清：
- 订单的事务性要求
- 查询模式
确认后再 propose。

## 案例3: /opsx:propose — 用户行为分析（SA ★ 主导，旧版）

**基于 explore 纪要**: 服务运营团队，回答"用户为什么流失"，Web+App，历史1年，T+1

**SA ★ 主导 — 方案设计**:

- 存储选型：ClickHouse（列存，适合海量行为分析）
- 采集方案：SDK 自研（Web+App 双端统一）
- 数据流：SDK → Kafka → Flink → ClickHouse → Grafana
- ADR-001: 选择 ClickHouse 而非 ES（理由：聚合查询性能更优）
- 备选方案A: ES+Kibana（全文搜索更强，聚合偏弱）
- 降级方案: MySQL+自建报表（数据量小时够用）

**PM 辅助 — 拆解任务**:
1. SDK 开发（Web端埋点 + App端埋点）
2. 数据管道搭建（Kafka + Flink）
3. 存储层部署（ClickHouse 集群）
4. 报表层开发（Grafana Dashboard）
5. 历史数据迁移
风险：历史数据格式不统一，迁移可能延期

**RR 辅助 — 准入评审**:
- 执行快速？✓ ClickHouse 聚合查询毫秒级
- 用户友好？✓ Grafana 可视化，运营自助查询
- 结论：准入通过

## 案例3: 旅游攻略方案设计

**BA 输出**: 9条需求 + 3条验收标准

**方案架构**:
- Day1: 出发→九江→庐山
- Day2: 九江→南昌
- Day3: 南昌→返程

**关键决策**:
- ADR-001: 庐山安排在Day1下午（而非Day2上午），因为Day2需要赶往南昌
- ADR-002: 选择高铁而非大巴（九江→南昌约1h vs 大巴3h）
- ADR-003: 九江住市区而非庐山上（方便Day2早出发去南昌）

**备选方案**:
- 主方案: 庐山半日+南昌两日
- 备选A: 庐山全日+南昌一日（南昌时间紧张）
- 降级方案: 仅九江或仅南昌（极端天气）

# PM 知识库 - 项目管理

## 核心方法论

### 任务拆解

- **WBS（工作分解结构）**：将复杂任务逐层分解为可管理的工作包
- **关键字拆解法**：从用户输入中提取关键字，每个关键字映射为一个子任务
- **MECE 原则**：拆解结果互斥且完备

### 进度管理

- **里程碑规划**：定义关键节点和交付物
- **关键路径法**：识别影响总工期的关键任务链
- **缓冲管理**：为高风险任务预留时间缓冲

### 风险管理

- **风险识别**：从需求、技术、资源、外部四个维度扫描
- **风险评估**：概率 × 影响 矩阵
- **风险应对**：规避 / 转移 / 减轻 / 接受
- **风险监控**：定期回顾和更新风险清单

### GitHub 同步策略

- **硬约束**：每个本地更新必须同步到 GitHub
- **PM 自主决策**：同步时机、粒度、分支策略
- **经验积累**：将同步策略的调整记录在 PM memory 中

### 项目归档

- **交付物盘点**：代码/文档/运维/监控等所有产出物
- **关键指标统计**：计划 vs 实际工期、偏差率、关键路径偏差原因
- **复盘要点**：做得好的（正面反馈）+ 教训（具体到"下次怎么做"）
- **可复用资产**：明确归属（哪个团队维护、存放位置）
- **待跟踪事项**：后续版本计划、持续运维关注点
- **教训反馈**：复盘教训必须反馈到对应 Agent 知识库

### 归档分级

| 项目规模 | 归档模板 | 必须项 |
|---------|---------|--------|
| 大型项目（多模块/跨团队） | 完整归档 | 交付物+指标+复盘+资产+跟踪 |
| 小型任务/紧急修复 | 轻量归档 | 时间线+Root Cause+教训+跟踪 |
| 任何规模 | 归档前检查 | TE 通过+生产验证+监控+文档 |

### 归档前检查

归档意味着"项目可移交、可遗忘"，不只是"代码合并了"。归档前必须确认：
- 所有 TE 验收用例通过
- 生产环境验证（至少 24h 或按项目要求）
- 监控告警已配置
- 运维文档已更新
- 如有未完成项，建议先回 /opsx:apply 补齐再归档

### 跨阶段共性原则

#### 阶段身份纪律

1. **不越界**：explore 阶段不出方案，propose 阶段不写代码，apply 阶段不改方案
2. **强衔接**：后一阶段的输出必须基于前一阶段的产出，缺前置必须回退
3. **可质疑**：任何角色发现上游产出有问题，必须主动提出
4. **门禁不可绕过**：RR 不通过的不能进 apply，CR 不通过的不能合并，TE 不通过的不能 archive

#### 输入识别

- 用户输入模糊 / 包含"调研""了解""看看" → explore
- 用户输入"设计""方案""怎么做" → propose
- 用户输入"实现""开发""修复""跑一下" → apply
- 用户输入"完成了""总结""复盘" → archive

如果用户用了错的命令，主动指出并建议正确阶段。

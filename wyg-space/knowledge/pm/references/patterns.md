# PM 模式 - 项目管理模式库

## 模式1: 关键字→子任务映射

| 关键字类型 | 映射规则 | 示例 |
|-----------|---------|------|
| 时间 | 提取时间约束，触发时间线规划 | "6月18日" → 确定出发日期 |
| 地点 | 提取地理约束，触发路线规划 | "九江" → 确定目的地范围 |
| 数量 | 提取量化约束，触发规模评估 | "三天两夜" → 行程时长 |
| 类型 | 提取任务类型，触发方法论选择 | "旅游" → 观光+休闲模式 |
| 交付物 | 提取期望产出，触发交付标准 | "攻略" → 可执行行程方案 |

## 模式2: 风险识别四维扫描

| 维度 | 典型风险 | 识别信号 |
|------|---------|---------|
| 需求 | 需求模糊、范围蔓延 | 关键字含义不明确 |
| 技术 | 技术不可行、依赖不可控 | 涉及外部系统/未知领域 |
| 资源 | 时间不足、能力缺口 | 约束与任务量不匹配 |
| 外部 | 天气/政策/市场变化 | 依赖外部因素 |

## 模式3: 回退频率预警

| 回退次数 | 预警级别 | 建议动作 |
|---------|---------|---------|
| 1-2次 | 正常 | 继续执行 |
| 3-4次 | 关注 | 检查是否需要调整方案 |
| 5+次 | 警告 | 可能需要回退到BA重新分析需求 |

## 模式4: 项目归档模式

```
1. 盘点交付物（代码/文档/运维/监控）
2. 统计关键指标（计划 vs 实际工期、偏差率）
3. 复盘（做得好的 + 教训）
4. 提炼可复用资产
5. 列出待跟踪事项
6. 执行归档操作（移动目录 + 同步 specs + Git 推送）
```

关键原则：
- 复盘要点必须具体，不能泛泛而谈
- 教训必须指明"下次怎么做"，而非只说"这次不好"
- 教训应反馈到对应 Agent 的知识库（BA/SA/RR/DEV/CR/TE）
- 可复用资产必须明确归属（哪个团队维护、存放位置）

## 模式4b: 轻量归档模式（小型任务/紧急修复）

```
1. 时间线（发现→修复→上线）
2. Root Cause
3. 关键教训（具体到"下次怎么做"）
4. 跟踪事项
```

与完整归档的区别：
- 省略交付物清单和关键指标统计
- 聚焦时间线 + Root Cause + 教训 + 跟踪
- 适用于紧急修复、小型优化等轻量场景

## 模式4c: 归档前检查模式

```
归档请求 → 检查归档前提件 → 全部通过 → 执行归档
                              → 存在未完成项 → 建议回 /opsx:apply 补齐
```

归档前检查清单：
- ✅ 所有 TE 验收用例通过
- ✅ 生产环境验证（至少 24h 或按项目要求）
- ✅ 监控告警已配置
- ✅ 运维文档已更新

关键原则：
- 归档意味着"项目可移交、可遗忘"，不只是"代码合并了"
- 未完成项应先回 /opsx:apply 补齐，而非带病归档

## 模式5: 复盘教训反馈模式

```
归档复盘 → 识别教训 → 判断教训归属 → 反馈到对应 Agent 知识库
```

| 教训类型 | 反馈目标 | 示例 |
|---------|---------|------|
| 需求遗漏 | BA explore 阶段 | "新中间件应让 OPS 介入" |
| 方案低估 | SA propose 阶段 | "ClickHouse 运维复杂度被低估" |
| 验收标准缺失 | RR 评审清单 | "漏了数据延迟指标" |
| 实现质量 | DEV 规范 | "弱网场景必须自测" |
| 评审遗漏 | CR 检查清单 | "线程安全检查项" |
| 测试覆盖 | TE 策略 | "弱网/断网场景纳入标准测试" |
